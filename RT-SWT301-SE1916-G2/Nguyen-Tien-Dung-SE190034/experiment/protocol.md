# Experiment Protocol — Gap-3 (LLM vs Manual vs EvoMaster on pre-seeded faults)

**Member:** Nguyen Tien Dung — SE190034 · **Topic:** SE1916 (LLM × REST API testing)
**Pre-registered design.** This protocol fixes the method *before* results are read. It operationalises
`SLR/gap-statement.md` (GAP-3), `experiment/01_rq.md` (RQ1–RQ3) and `experiment/hypotheses.md` (H0/H1).

---

## 1. Objective

Close GAP-3 (and, jointly, GAP-1/GAP-2): on locally deployed EvoMaster-Benchmark (EMB) APIs with
**pre-seeded faults of known ground truth**, measure on one common footing:

- **RQ1 (coverage):** LLM endpoint coverage vs the ≥90% target + which endpoint types are missed.
- **RQ2 (fault detection):** fault-detection **recall** of **LLM vs Manual vs EvoMaster** on the same
  seeded-fault set — the comparison no prior paper has made (they only *count* 500s without ground truth).
- **RQ3 (edge cases):** edge-case / error-code scenarios per endpoint, **LLM vs Manual**.

## 2. Systems Under Test (SUTs)

Three REST APIs from the **EvoMaster Benchmark (EMB)** `jdk_8_maven/cs/rest`, built with **JDK 8**
(Temurin 1.8.0_492), deployed locally as standalone Spring-Boot fat-jars.

| SUT | EMB path | Ops | Endpoint character | DB | Port |
|-----|----------|----:|--------------------|----|----:|
| `rest-ncs` | artificial/ncs | 6 | numeric algorithms, input-validation/boundary (400s **undocumented** in spec) | none | 8080 |
| `rest-scs` | artificial/scs | 11 | string algorithms, validation/boundary | none | 8083 |
| `features-service` | original/features-service | 18 | **CRUD** + feature/configuration/constraint resources (cited in EvoMaster #10, DeepREST #13) | embedded | 8081 |

Specs frozen under `experiment/specs/*.openapi.json` (sha256 recorded). The springfox-generated specs
document only 200/401/403/404 while the implementations also return 400 on out-of-range inputs — the
**spec≠implementation gap** RQ3 targets. (user-management was evaluated and dropped: it requires an
external MySQL; out of scope for a self-contained local run. Recorded as a threat / future SUT.)

## 4. The three arms (Intervention + Comparators)

All three arms are **black-box HTTP test suites** in one common harness (`experiment/harness/`,
packages `llm` / `manual` / `evomaster`), each test driven by REST-assured against a configurable
`baseURI`. This uniform footing is what makes the 3-way comparison fair.

- **I — LLM** (`experiment/harness/.../llm/`): generated by **Claude Sonnet 4.6** sub-agents with a
  *clean context*, given ONLY the OpenAPI spec + a frozen prompt (`experiment/llm/prompt_template.md`).
  **Blind to source and faults** (tool-use logs in `experiment/llm/transcripts/` show 0–1 tool calls:
  spec read only). Verbatim model output, committed unedited.
- **C1 — Manual** (`.../manual/`): designed by a sub-agent acting as a human QA engineer using a
  conventional methodology (equivalence partitioning + boundary-value analysis + error guessing +
  CRUD lifecycle), spec-only, blind. See `experiment/manual/methodology.md` — including the disclosed
  construct-validity threat that both arms are Claude-authored (a human cohort is future work).
- **C2 — EvoMaster** (`.../evomaster/`): EvoMaster 6.0.0 black-box, `--maxTime 120s`, output
  `JAVA_JUNIT_5`, adapted into the harness (package + configurable baseURI; EvoMaster's helper
  matchers reproduced faithfully as committed shims — see `experiment/evomaster/README.md`).
  Note: EvoMaster records **exact response values** as regression oracles (`numberMatches(...)`),
  unlike the spec-derived status oracles of LLM/Manual — a methodological difference reported, not hidden.

**Blind protocol:** all three suites are produced **before** faults are seeded; the seeding tool never
sees the suites and the suites never see the mutants.

## 5. Fault model — system-level mutation (RQ2 ground truth)

Faults are seeded with **standard mutation operators** (the families PIT / Offutt use: relational-operator
replacement, arithmetic-operator replacement, negate/boundary conditionals) applied **systematically**
to each SUT's **REST-controller / resource layer**, where a fault is **observable through the HTTP
contract** — the relevant fault population for black-box API testing. Generator: `experiment/scripts/mutate.py`.

- Each candidate mutation → the SUT is **recompiled** into a standalone mutant jar; only mutants that
  **compile** are kept (non-compiling discarded — standard). Catalog (id, file, line, operator,
  before→after) in `experiment/faults/<sut>/catalog.json` = the **ground truth** (known total faults).
- **Detection (kill):** a suite *kills* a mutant if a test that **passes on the original** SUT **fails**
  on the mutant (behaviour observably changed). Tests failing on the original are excluded as oracles.
- **Recall** = killed / total mutants, per arm per SUT. (This is the ground-truth Recall the SLR found
  missing everywhere — `evidence-table.md` §D.)

## 6. Measurement

- **RQ1 — endpoint coverage (LLM):** fraction of spec operations exercised by ≥1 request, per SUT;
  compared to the 0.90 target; plus a breakdown by endpoint type (CRUD / validation / error-handling)
  of which are least covered. EvoMaster's own coverage report cross-checks. Code coverage (JaCoCo)
  reported as secondary where available.
- **RQ2 — fault recall:** §5, all three arms on the identical mutant set.
- **RQ3 — edge-case scenario count (LLM vs Manual):** count of distinct `negative`+`boundary`+`errorcode`
  scenarios per endpoint, parsed from the committed `// SCENARIO ... type=...` tags, with a
  4xx/5xx/boundary breakdown.

## 7. Statistics (α = 0.05, per `hypotheses.md`)

| RQ | Test |
|----|------|
| RQ1 | one-sample **Wilcoxon signed-rank** of per-operation/per-SUT coverage vs 0.90 |
| RQ2 | **Friedman** (3 arms × 3 SUTs) + post-hoc **Wilcoxon** (Holm) + per-mutant **McNemar** (pooled), effect sizes |
| RQ3 | paired **Wilcoxon signed-rank** across endpoints (LLM vs Manual counts) |

`experiment/scripts/analyze.py` (scipy/statsmodels) consumes the raw per-test / per-mutant CSVs in
`experiment/results/raw/` and emits tables + figures + p-values to `experiment/results/`.

## 8. Integrity guards (why the result is hard to refute)

1. **Real execution only** — every number comes from real HTTP calls / real Maven builds / real EvoMaster
   runs; no figure is hand-entered. Raw surefire XML + mutant catalogs + EvoMaster reports are committed.
2. **Blindness** — suites authored before/independent of fault seeding; LLM/Manual blind to source
   (tool logs prove it); the same mutant set scores all arms.
3. **Reproducibility** — pinned tool versions, frozen specs (sha256), committed prompt template,
   committed scripts + seeds; `README.md` reproduces end-to-end from an EMB checkout.
4. **Honest oracles** — baseline-failing tests excluded; mutants that change no observable behaviour are
   reported as such; the EvoMaster regression-oracle difference is disclosed.

## 9. Threats to validity (disclosed)

- **Construct:** LLM and Manual arms share Claude as author (different role/methodology); a human-tester
  cohort would strengthen "LLM vs human". Mitigated by distinct documented methodologies + blindness.
- **External:** 3 small EMB APIs; results may not generalise to large industrial APIs. n = 3 limits the
  Friedman power → per-mutant McNemar (pooled) reported as the primary RQ2 test.
- **Internal:** LLM non-determinism (a re-run yields a similar-but-not-identical suite — disclosed, raw
  output frozen); mutation operators scoped to the controller layer (observable faults).
- **Conclusion:** small samples → non-parametric tests + effect sizes, not just p-values.
