## 1. Executive summary

We compare three REST-API test-generation "arms" — **Claude Sonnet 4.6 (LLM)**, **Manual (Equivalence Partitioning + Boundary Value Analysis)**, and **EvoMaster 6.0.0** — on **3 EvoMaster-Benchmark (EMB) REST services** totalling **35 operations**, using **133 systematically pre-seeded mutation faults** as ground truth. Three research questions are answered under a pre-registered protocol (α = 0.05):

| RQ | What it measures | Headline result | Statistical verdict |
|----|------------------|-----------------|---------------------|
| **RQ1** | LLM endpoint coverage vs the ≥90% target | LLM reaches **100%** (35/35 operations) | **Reject H0** (Wilcoxon W=630, p ≈ 1.6×10⁻⁹) |
| **RQ2** | Fault-detection Recall: LLM vs Manual vs EvoMaster | LLM **0.0602** (8/133) > Manual **0.0150** (2/133); EvoMaster **0.1353** (18/133) highest | **Fail to reject H0** (LLM not > *both*: EvoMaster beats it, McNemar p ≈ 0.031) |
| **RQ3** | Edge-case scenarios per endpoint: LLM vs Manual | LLM **217** vs Manual **113** (median 5 vs 3) | **Reject H0** (Wilcoxon W=626, p ≈ 1.4×10⁻⁷) |

**Overall verdict.** The LLM saturates endpoint coverage and generates significantly more edge-case scenarios than manual design. On ground-truth fault detection it **significantly out-detects an independent Manual suite** (Recall 0.0602 vs 0.0150, McNemar p ≈ 0.031) but is itself **out-detected by EvoMaster** (0.1353, p ≈ 0.031) — so it is not superior to *both* baselines and the RQ2 hypothesis (H1: LLM > both) is not supported. Note: the Week-7 pilot showed an exact LLM = Manual tie; §7 shows this was an artifact of same-agent authorship that broke once the Manual suite was regenerated by an independent agent. The reason no status-oracle arm approaches EvoMaster is explained mechanistically in §7 (a lenient, existence-based oracle that is blind to silent value faults).

## 2. Research profile

**Problem.** A 13-paper systematic literature review (`SLR/`, N=13) found that 2019–2026 work applies LLMs to REST-API test generation and reports endpoint coverage and raw 500-error *counts*, but leaves three gaps:

- **GAP-1** — no study compares **LLM vs Manual vs EvoMaster** on the *same* APIs.
- **GAP-2** — no study uses a **shared pre-seeded-fault** dataset, so fault-detection **Recall** is never computed.
- **GAP-3 (our focus, GAP-M "Metric")** — no study reports a **per-endpoint edge-case / error-code metric**, nor an endpoint-type miss analysis.

**Goal.** On locally deployed EMB APIs instrumented with pre-seeded faults of known ground truth, measure the three arms' effectiveness on one common footing, answering RQ1–RQ3.

## 4. Method (condensed from `protocol.md`)

**Systems under test (EMB, JDK 8).**

| SUT | Operations | Domain focus | Pre-seeded mutants (ground truth) |
|-----|-----------:|--------------|----------------------------------:|
| `rest-ncs` | 6 | numeric / boundary | 70 |
| `rest-scs` | 11 | string / boundary | 59 |
| `features-service` | 18 | CRUD + constraints | 4 |
| **Total** | **35** | — | **133** |

**Arms (uniform black-box REST-assured harness).**
- **I = LLM** — Claude Sonnet 4.6, blind, frozen prompt, generated from the OpenAPI spec only.
- **C1 = Manual** — blind EP/BVA test design.
- **C2 = EvoMaster 6.0.0** — black-box search-based generation.

All suites were produced **before** faults were seeded (blind protocol).

**Faults (ground truth).** Standard mutation operators (relational / arithmetic / negate-boundary — the PIT/Offutt families) were seeded systematically into the controller layer; each mutant was recompiled into a standalone SUT; only compilable mutants were kept. Catalog = `faults/<sut>/catalog.json`.

**Kill rule.** An arm *kills* a mutant iff a test that **passes on the original** SUT **fails on the mutant**. **Recall = killed / total mutants.**

**Metrics.** RQ1 = endpoint coverage (% of operations exercised) vs a 90% target, plus endpoint-type miss analysis. RQ2 = mutation-kill Recall per arm. RQ3 = edge-case scenario count (negative + boundary + error-code) per endpoint.

**Statistics (α = 0.05, pre-registered in `hypotheses.md`).** RQ1: one-sample Wilcoxon signed-rank vs 0.90. RQ2: Friedman omnibus + post-hoc pairwise Wilcoxon (Holm-corrected) + pooled per-mutant McNemar + Cliff's δ. RQ3: paired Wilcoxon signed-rank (LLM vs Manual per endpoint) + rank-biserial. A stricter Bonferroni threshold across the 3 RQ families (α_adj ≈ 0.0167) is additionally reported for transparency.

## 5. Results

### 5.1 Suite sizes (tests generated and executed on the real SUTs)

| SUT | LLM | Manual | EvoMaster |
|-----|----:|-------:|----------:|
| `rest-ncs` | 86 | 51 | 30 |
| `rest-scs` | 109 | 54 | 27 |
| `features-service` | 100 | 42 | 30 |
| **Total** | **295** | **147** | **87** |

*Source: `results/stats/master_summary.csv`. The Manual suite is the Week-8 independent-agent regeneration (spec-only, blind to faults and to the LLM output); it is leaner than the pilot's same-session Manual suite.*

### 5.2 RQ1 — Endpoint coverage of the LLM suite (vs ≥90% target)

The LLM suite exercises **35/35 operations = 100%** endpoint coverage (ncs 6/6, scs 11/11, features 18/18). One-sample Wilcoxon of per-operation coverage against 0.90: **W = 630, p ≈ 1.6×10⁻⁹ < 0.05 → reject H0**; rank-biserial = 1.0. LLM endpoint coverage significantly exceeds the 90% target.

**Endpoint-type miss analysis.** With operation-level coverage saturated at 100%, *no* endpoint type (numeric-validation / string-validation / CRUD / constraint) is missed at the routing level. The gap therefore is not *which* endpoints are reached, but *what is asserted* once they are reached — which is exactly what RQ2 and RQ3 probe.

*Figure: `results/figures/rq1_coverage.png`.*

### 5.3 RQ2 — Fault-detection Recall: LLM vs Manual vs EvoMaster

Pre-seeded mutants (ground truth): ncs = 70, scs = 59, features = 4 (total 133).

| SUT | LLM Recall | Manual Recall | EvoMaster Recall | (killed / mutants) |
|-----|-----------:|--------------:|-----------------:|--------------------|
| `rest-ncs` | 0.0000 | 0.0000 | 0.1714 | LLM 0/70 · Man 0/70 · Evo 12/70 |
| `rest-scs` | 0.1186 | 0.0169 | 0.0847 | LLM 7/59 · Man 1/59 · Evo 5/59 |
| `features-service` | 0.2500 | 0.2500 | 0.2500 | LLM 1/4 · Man 1/4 · Evo 1/4 |
| **Overall** | **0.0602** | **0.0150** | **0.1353** | LLM 8/133 · Man 2/133 · Evo 18/133 |

*Source: `results/stats/rq2_recall_per_sut.csv` + `results/stats/summary.json` (Week-8 full run).*

**Tests.**
- **Friedman** (3 arms × 3 SUTs): χ² = 2.0, p = 0.368 — no omnibus difference detectable with only n = 3 SUTs (the per-SUT Recall test is underpowered; the pooled per-mutant McNemar below is the primary RQ2 test).
- **Post-hoc pairwise Wilcoxon (Holm-corrected):** LLM vs Manual p = 1.0; LLM vs EvoMaster p = 1.0 (per-SUT recall vectors, n = 3 — non-significant purely from the tiny sample).
- **Pooled per-mutant McNemar (primary, N = 133):** LLM vs Manual — b = 6 (LLM-only kills), c = 0 (Manual-only kills), statistic = 0.0, **p ≈ 0.031 < 0.05 → reject H0: the LLM kills strictly more mutants than the independent Manual suite** (every mutant Manual kills, LLM also kills). LLM vs EvoMaster — b = 4 (LLM-only), c = 14 (EvoMaster-only), statistic = 4.0, **p ≈ 0.031 < 0.05 → reject H0: EvoMaster kills strictly more than the LLM**; approximate achieved power ≈ 0.46–0.56 (disclosed as an approximation).
- **Cliff's δ:** LLM vs Manual = +0.045 (small, favouring LLM); LLM vs EvoMaster = −0.075 (small, *against* the LLM).

**Verdict.** **The RQ2 hypothesis (H1: LLM Recall > *both* baselines) is not supported → H0 stands**, because EvoMaster out-detects the LLM (0.1353 vs 0.0602, McNemar p ≈ 0.031). The LLM does, however, significantly out-detect the **independent** Manual suite (0.0602 vs 0.0150, p ≈ 0.031, Manual's kills ⊂ LLM's kills). The exact LLM = Manual tie seen in the Week-7 pilot has disappeared once the Manual suite was authored by an independent agent — evidence that the pilot tie was an authorship artifact, not a property of the arms (see §7). The study remains underpowered at the SUT level (n = 3), so the per-mutant McNemar carries the conclusion.

*Figure: `results/figures/rq2_recall.png` and `figures/fig1_distribution.png`.*

### 5.4 RQ3 — Edge-case / error-code scenarios per endpoint: LLM vs Manual

The LLM suite produces **217** edge-case scenarios (negative + boundary + error-code) versus the independent Manual **113** across the 35 endpoints; median **5 vs 3** per endpoint. Paired Wilcoxon (LLM − Manual per endpoint): **W = 626, p ≈ 1.4×10⁻⁷ < 0.05 → reject H0**; rank-biserial = 0.987. The LLM produced strictly more edge-case scenarios on **33/35** endpoints, fewer on **2**, and tied on **0**.

**Verdict.** **H0 is rejected.** The LLM is significantly more prolific at generating error-code / boundary scenarios than the manual EP/BVA methodology.

*Figure: `results/figures/rq3_edgecases.png` and `figures/fig2_comparison.png`.*

### 5.5 Multiplicity

Per-RQ conclusions above use the pre-registered raw α = 0.05. Under a stricter Bonferroni correction across the 3 RQ families (α_adj ≈ 0.0167), RQ1 (p ≈ 1.6×10⁻⁹) and RQ3 (p ≈ 1.4×10⁻⁷) remain significant. RQ2's two McNemar comparisons (both p ≈ 0.031) are significant at the raw α = 0.05 but **not** under the stricter Bonferroni α_adj ≈ 0.0167 — disclosed transparently. The headline RQ2 conclusion (LLM does not exceed *both* baselines, since EvoMaster leads) is unchanged either way, as is the direction of every effect; only the borderline significance of the pairwise gaps is threshold-sensitive.

## 6. Answers to the research questions

- **RQ1 — "Do LLM tests achieve ≥90% endpoint coverage?"** → **Yes.** 100% coverage, significantly above 90% (p ≈ 1.6×10⁻⁹). No endpoint type is missed at the routing level.
- **RQ2 — "Do LLM tests detect more pre-seeded faults than Manual *and* EvoMaster?"** → **No (not better than both).** The LLM significantly out-detects the independent Manual suite (Recall 0.0602 vs 0.0150, McNemar p ≈ 0.031) but is itself out-detected by EvoMaster (0.1353, p ≈ 0.031). Since it does not beat *both* baselines, H0 stands.
- **RQ3 — "Are LLM tests more prolific at edge-case / error-code scenarios than Manual?"** → **Yes.** 217 vs 113 scenarios, more on 33/35 endpoints (p ≈ 1.4×10⁻⁷).

## 7. Threats to validity

**Construct (oracle strength — the dominant effect).** All three status-oracle arms have low absolute Recall because the LLM and Manual suites assert **existence, not value** (e.g. `body("resultAsDouble", notNullValue())`, `hasKey(...)`), so they are structurally blind to *silent arithmetic* mutants on `rest-ncs` and `rest-scs` — on ncs (all value faults) every status-oracle suite kills 0/70 while EvoMaster's recorded-value regression oracle kills 12. This is `protocol.md`/proposal §7's pre-registered "lenient oracle" threat, **empirically confirmed** and reported as a finding, not corrected post-hoc (which would be HARKing). The mechanism is two-tier: (1) the **kind** of oracle sets the ceiling (status vs recorded-value); (2) within the same status-oracle regime, suite **volume** then decides — on scs the larger LLM suite kills 7 vs the leaner independent Manual's 1 (Manual's kills ⊂ LLM's). EvoMaster's higher Recall reflects tier (1), a disclosed oracle difference (`evomaster/README.md`).

**Same-author baseline (pilot artifact, now resolved).** In the Week-7 pilot the LLM and Manual arms were authored in the same agent session and killed an identical set of mutants on all 133 — an exact tie. The RBL-4 Week-8 design (`RBL4-FULL-RUN-DESIGN.md` §3; run log `notes.md` §F0) re-authored the Manual suite with an **independent, isolated agent** (spec-only, blind to the fault catalog and to the LLM output), disclosed as *independent-agent*, not *independent-human*. This report reflects that Week-8 run: the tie broke (scs Manual fell 7→1 kills), showing the pilot's LLM = Manual identity was an authorship artifact rather than a property of the methods. This is the primary threat that motivated the full run, and it has been addressed.

**External.** Only 3 small EMB APIs (n = 3) limits Friedman power, so the pooled per-mutant McNemar is the primary RQ2 test. The `user-management` (MySQL) service was excluded, so authentication endpoints are under-represented. `features-service` carries only **4** seeded mutants versus 70 (ncs) / 59 (scs) — a real pre-existing imbalance from the Week-7 mutation run; it is reported honestly as a generalisation limit and **not** patched by adding data post-hoc.

**Internal.** Mutations were seeded in the controller layer (observable faults). LLM output is non-deterministic; the raw generated suites are frozen and committed for reproducibility.

**Conclusion validity.** Given small samples, non-parametric tests and effect sizes (Cliff's δ, rank-biserial) are reported alongside p-values, and approximate achieved power is disclosed for the RQ2 comparison.

## 8. Conclusion

On a shared pre-seeded-fault benchmark, an LLM (Claude Sonnet 4.6) **reaches 100% endpoint coverage** (well above the 90% target) and **generates significantly more edge-case scenarios** than manual EP/BVA design, confirming the GAP-M contribution that the SLR identified as missing. However, breadth of scenarios does **not** make it the best fault detector: on ground-truth Recall the LLM **significantly out-detects an independent Manual suite (0.0602 vs 0.0150) but is itself beaten by EvoMaster (0.1353)**, so it is not superior to both baselines and the RQ2 hypothesis fails. The limiting factor is oracle strength, not endpoint reach or scenario volume — a suite that asserts response *existence* rather than response *value* misses silent faults no matter how many scenarios it enumerates (on `rest-ncs` every status-oracle arm scores 0). Where oracle kind is held constant, suite volume is what separates the arms, which is why the more prolific LLM beats the leaner Manual suite. The practical implication for LLM-assisted API testing is that prompting must target **assertion strength / value-oracles**, not just scenario coverage.
