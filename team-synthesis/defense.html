Vì sao test REST API lại quan trọng & tốn?

REST API = cách các phần mềm "nói chuyện" với nhau (app ↔ server).

Ở khắp nơi

Mọi app mobile/web đều gọi REST API. Một lỗi API = nhiều app hỏng theo.

Test rất tốn

Mỗi endpoint phải thử: input đúng, input sai, giá trị biên, lỗi… Viết tay rất lâu.

Ví von

Như đi kiểm tra mọi cánh cửa của một toà nhà: đúng khoá, sai khoá, cửa kẹt, cửa cấm…

→ Nếu có công cụ tự sinh test tốt, tiết kiệm cực nhiều công sức QA.

🎤 Nói: "Trước hết, REST API là cách các phần mềm nói chuyện với nhau — app gọi server qua API. Nó có ở khắp nơi, nên một lỗi API kéo theo nhiều app hỏng. Mà test API thì rất tốn: mỗi endpoint phải thử input đúng, input sai, giá trị biên, các trường hợp lỗi. Ví von như phải đi kiểm tra mọi cánh cửa của một toà nhà. Vì vậy, nếu có công cụ tự sinh test tốt thì tiết kiệm rất nhiều."
01

AI (LLM) làm gì trong bài này?

Cho AI đọc bản mô tả API (OpenAPI spec) → AI tự viết ra code test.

Đầu vào: OpenAPI spec
GET /api/bessj/{n}/{x} trả về: số (JSON) n: integer, x: number
AI (LLM) suy luận
"n là số nguyên → thử n âm, n=0, n rất lớn, thử x sai kiểu…"
Đầu ra: test tự động
@Test bessj_n1() get("/api/bessj/1/2.0") .statusCode(anyOf(200,400))

AI chỉ nhìn spec, không nhìn code nguồn (black-box) — y như một tester thật ngoài đời.

Quan trọng cho phần bảo vệ: AI "mù" với code → công bằng, không gian lận.
🎤 Nói: "AI ở đây làm việc rất cụ thể: tụi em đưa cho nó bản mô tả API — gọi là OpenAPI spec — rồi nó tự viết ra code test. Ví dụ spec nói endpoint bessj nhận một số nguyên n; AI suy luận 'vậy thử n âm, n bằng 0, n rất lớn' rồi sinh test tương ứng. Điểm mấu chốt: AI chỉ nhìn spec, không nhìn code nguồn — gọi là black-box — đúng như một tester thật. Điều này làm thí nghiệm công bằng."

Bước 3 — "Gieo lỗi" (mutation) là gì?

Cố ý sửa nhẹ code để tạo một con bug — rồi xem test nào phát hiện.

Code gốc (đúng)
if (n < 2) return 400;
Mutant (đã gieo lỗi)
if (n <= 2) return 400; // đổi < thành <=
Hậu quả
n = 2: gốc → 200 (đúng) mutant → 400 (SAI!)

Test "giết" được mutant nếu: pass trên code gốc nhưng fail trên mutant.

Recall = số mutant bị giết / tổng mutant. Pilot: 133 mutant (đáp án biết trước).

🎤 Nói: "Bước ba là phần hay nhất — gieo lỗi. Tụi em cố ý sửa nhẹ code để tạo một con bug. Ví dụ code gốc kiểm tra 'nếu n nhỏ hơn 2 thì báo lỗi'; tụi em đổi thành 'nhỏ hơn hoặc bằng 2'. Giờ với n bằng 2, code gốc trả đúng nhưng mutant trả sai. Một test được tính là 'giết' mutant nếu nó pass trên code gốc nhưng fail trên mutant. Recall là số mutant giết được chia tổng mutant. Pilot tụi em gieo 133 mutant, biết trước đáp án."

Bước 4 — Đo 3 thứ

Mỗi thước đo trả lời một câu hỏi khác nhau.

→ RQ1

Coverage

% endpoint được ít nhất 1 test chạm tới. "Có gọi tới chưa?"

→ RQ2

Recall lỗi gieo

% mutant bị giết. "Bắt được lỗi thật không?"

→ RQ3

Edge-case

Số kịch bản negative/boundary/errorcode mỗi endpoint. "Test có kỹ không?"

3 thước đo cùng nhau = bức tranh đầy đủ: rộng (coverage) + sâu (edge-case) + thật (Recall).

🎤 Nói: "Bước bốn, đo ba thứ vì mỗi thứ trả lời một câu khác nhau. Coverage là phần trăm endpoint được chạm tới — trả lời 'có gọi tới chưa'. Recall là phần trăm mutant bị giết — 'có bắt được lỗi thật không'. Edge-case là số kịch bản lỗi và biên cho mỗi endpoint — 'test có kỹ không'. Ba thước đo cùng lúc cho bức tranh đầy đủ: rộng, sâu, và thật — chứ không chỉ nhìn coverage như các paper cũ."

Ví dụ chạy thật: endpoint bessj

Đây là "cái bẫy" mà nghiên cứu soi ra — rất dễ kể khi bảo vệ.

Spec KHAI BÁO
GET /api/bessj/{n}/{x} responses: 200, 401, 403, 404 // KHÔNG có 400!
CODE thật làm
if (n < 2 || n > 1000) return 400; // NcsRest.java:43
Bẫy
bessj/1/2.0 → 400 nhưng spec không ghi 400 → ai chỉ đọc spec sẽ KHÔNG biết bẫy này

→ Test giỏi phải tự đoán "thử n=1". Pilot: AI bắt được bẫy này, một số test người bỏ sót. NcsRest.java:43

Nếu GV hỏi "Gap-M cụ thể là gì?" → kể đúng ví dụ này.
🎤 Nói: "Đây là ví dụ cụ thể nhất, rất dễ kể. Endpoint bessj: bản mô tả spec chỉ khai báo các mã 200, 401, 403, 404 — không có 400. Nhưng code thật lại có dòng 'nếu n nhỏ hơn 2 thì trả 400'. Nghĩa là gọi bessj với n=1 sẽ ra lỗi 400 — một cái bẫy mà spec không hề ghi. Ai chỉ đọc spec sẽ không biết. Test giỏi phải tự đoán 'thử n bằng 1'. REST API — cách phần mềm gọi nhau qua HTTP (GET/POST…).
OpenAPI spec — bản mô tả API (endpoint, input, output) = "hợp đồng".
LLM — mô hình ngôn ngữ lớn (AI như GPT/Claude) đọc & viết text.
SLR — đọc tài liệu có hệ thống để tìm khoảng trống.
GAP — khoảng trống nghiên cứu (cái quan trọng chưa ai làm).
RQ — câu hỏi nghiên cứu (cái ta đi trả lời).
H0 / H1 — giả thuyết "không khác biệt" / "điều muốn chứng minh".
p-value — xác suất kết quả do may rủi; < 0.05 = đáng tin.
Baseline — mốc/đối thủ để so (Người, EvoMaster).
SOTA — state-of-the-art = tốt nhất hiện có.

Giải ngố — thuật ngữ Experiment

Các từ kỹ thuật trong phần chạy thực nghiệm.

Black-box — test chỉ nhìn input/output, KHÔNG nhìn code.
EP / BVA — chia lớp tương đương / thử giá trị biên (test tay).
Mutation / mutant — cố ý sửa code tạo "bug giả" để thử test.
Kill (giết mutant) — test pass trên code gốc nhưng fail trên mutant.
Recall — % lỗi gieo mà test bắt được.
Coverage — % endpoint được test chạm tới.
Edge-case — kịch bản biên/lỗi (4xx/5xx, giá trị cực trị).
Oracle — "đáp án" test dùng để biết đúng/sai (status / giá trị).
EvoMaster — tool tự sinh test bằng thuật toán tiến hoá.
Wilcoxon / McNemar — kiểm định thống kê phi tham số (so cặp).
