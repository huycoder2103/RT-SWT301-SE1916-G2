Nền tảng · 1/5

REST API là gì — và "spec" OpenAPI

  • REST API = cách hai phần mềm nói chuyện qua HTTP. Mỗi tài nguyên (user, order…) có một URL; thao tác bằng method: GET đọc, POST tạo, PUT/PATCH sửa, DELETE xoá.
  • Server trả về status code: 2xx thành công · 4xx lỗi phía client (sai input, thiếu quyền) · 5xx lỗi phía server (crash).
  • OpenAPI / Swagger = file đặc tả (YAML/JSON) mô tả: có những endpoint nào, tham số gì, kiểu dữ liệu, schema request/response, và mô tả bằng ngôn ngữ tự nhiên.
  • Đây là đầu vào chuẩn cho hầu hết công cụ sinh test — 11/13 paper đọc trực tiếp OpenAPI.
VÍ DỤ — một endpoint trong OpenAPI
paths: /pets/{id}: get: parameters: - name: id in: path required: true schema: { type: integer, minimum: 1 } responses: '200': { description: OK } '404': { description: Not found }
!
Vì sao quan trọng: dòng minimum: 1 và mã 404 chính là chỗ để sinh edge case — gửi id = 0 hay id không tồn tại để buộc 4xx. Cả bài sẽ xoáy vào việc ai sinh được những edge case này.

Mục tiêu slide: đảm bảo cả hội đồng (kể cả người không chuyên backend) hiểu 3 khái niệm: method, status code, OpenAPI spec.

Nhấn vào ví dụ: chỉ vào minimum: 1 và 404. Nói: "Một bộ test tốt không chỉ gọi GET hợp lệ, mà phải cố tình vi phạm ràng buộc này — gửi id=0, gửi id chữ — để xem API có trả đúng 4xx không. Đây là edge case, và là trục chính của RQ3 sau này."

Chốt: "OpenAPI là đầu vào chuẩn — 11/13 paper dùng nó. Giữ ý này, lát nữa nó là chữ P và I trong PICO."

Nền tảng · 2/5

Sinh test REST API tự động — khó ở đâu?

Thách thức 1

Không gian đầu vào vô hạn

Mỗi tham số có vô số giá trị hợp lệ/không hợp lệ. Phải sinh giá trị thực tế (email đúng định dạng, ngày hợp lệ) thì request mới qua được 4xx.
Thách thức 2

Phụ thuộc & trạng thái

Phải POST tạo resource rồi mới GET được nó. Thứ tự lời gọi (stateful) quyết định có chạm tới endpoint sâu hay không.
Thách thức 3

Bài toán Oracle

Làm sao biết kết quả đúng? Spec ≠ code thật. Nhiều tool chỉ coi 5xx = lỗi → bỏ sót lỗi logic trả 200 nhưng sai.
!
Hai loại "độ phủ" (coverage) — đừng nhầm: (a) Endpoint/operation coverage = đã gọi bao nhiêu % endpoint; (b) Code coverage = đã chạy bao nhiêu % dòng/nhánh code server. Loại (a) có thể rất cao; loại (b) thường thấp hơn nhiều. Sự khác biệt này là chìa khoá của RQ1.

3 thách thức giải thích vì sao đây là bài toán nghiên cứu mở chứ không phải "viết vài test là xong".

Thách thức 1 (giá trị thực tế) → đây đúng là chỗ LLM mạnh, nhớ để nối sang slide sau. Thách thức 3 (oracle) → giải thích vì sao đo "fault detection" rất khó và hầu hết paper chỉ đếm 5xx.

Cực kỳ quan trọng — nói chậm: phân biệt 2 loại coverage. Hội đồng rất hay hỏi "90% coverage là nhiều mà, sao còn gap?". Trả lời: 90% là endpoint coverage (gọi tới endpoint), còn code coverage chỉ ~52%. RQ1 của em đo endpoint coverage và còn bóc tách loại endpoint nào hay bị bỏ sót — cái chưa ai làm.

Nền tảng · 3/5

Trước LLM: 4 trường phái sinh test tự động

Search-based

EvoMaster 2019

Thuật toán di truyền (genetic / MIO) tối ưu test suite theo code coverage. White-box (đọc cả mã nguồn). Là baseline được tái sử dụng nhiều nhất trong cả corpus.
Model-based

Morest 2022

Dựng "Property Graph" của API rồi cập nhật bằng phản hồi thực thi để chọn chuỗi lời gọi hợp lý.
Fuzzing có trạng thái

RESTler 2019

Phân tích spec, sinh chuỗi request có trạng thái để dò 5xx. Nền tảng cho fuzzing REST.
Reinforcement learning

DeepREST 2024

Deep RL + "tò mò" (curiosity) học các ràng buộc ngầm ngoài spec. Mạnh nhưng cần huấn luyện.
!
Giới hạn chung (sẽ chứng minh bằng số ở Phần 3): dù thông minh, nhóm này bị chặn ở ~52% line coverage trên 20 service chuẩn, và EvoMaster còn sinh test phủ THẤP hơn người viết tay (41% vs 82%). Đó là lằn ranh mà LLM hứa hẹn vượt qua.

4 trường phái này chính là 4 baseline non-LLM (#10–#13) sẽ gặp ở Phần 3. Giới thiệu trước để khi tới đó hội đồng đã quen tên.

Nhấn 2 con số "treo" để tạo tò mò: 52% line coverage là trần (paper #11), và EvoMaster thua test tay 41% vs 82% (paper #10). Hai con số này là lý do người ta kỳ vọng LLM, và cũng là baseline để so trong thí nghiệm.

EvoMaster nhấn mạnh: "đây là tool em chọn làm baseline tự động trong thí nghiệm, vì nó được tái dùng nhiều nhất."

Nền tảng · 4/5

LLM bước vào — nó làm được gì mà tool cũ không?

  • Đọc được mô tả ngôn ngữ tự nhiên trong spec. Tool dựa luật (rule-based) không hiểu câu "id must be a valid ISO country code"; LLM thì hiểu → sinh đúng giá trị. (đây là lý do RESTGPT đạt 97% precision trích luật — Phần 3)
  • Sinh giá trị đầu vào "giống thật". Email, ngày tháng, mã định danh hợp lệ → vượt qua tầng kiểm tra 4xx để chạm code sâu hơn.
  • Suy luận phụ thuộc giữa các operation. Biết phải tạo trước rồi mới truy vấn — nhờ "đọc hiểu" cả tài liệu.
  • Sinh thẳng script test chạy được (xUnit, Postman, Karate) và cả kịch bản âm (negative cases).
?
Nhưng có cái giá: LLM ảo giác (hallucination), chi phí token, và oracle vẫn không chắc. Vì vậy câu hỏi không phải "LLM có dùng được không" mà là "hiệu quả tới mức nào, so với người và so với EvoMaster, trên thước đo có sự thật nền (ground truth)?" — đúng tinh thần SLR.

Đây là bản lề: chuyển từ "tool cũ" sang "vì sao LLM". 4 năng lực — mỗi cái nối tới một paper cụ thể ở Phần 3 (RESTGPT đọc NL, LlamaRestTest sinh giá trị, KAT suy luận dependency, APITestGenie sinh script).

Câu callout cuối là framing của cả bài: không tung hô LLM, mà đặt câu hỏi có kiểm soát. Nói đúng câu in đậm — nó dẫn thẳng tới Main RQ.

Từ PICO ra 2 chuỗi tìm kiếm

STRING 1 — LLM sinh test REST API (chính)
("REST API" OR "Web API" OR OpenAPI OR Swagger) AND ("test generation" OR "test case generation" OR "automated testing") AND (LLM OR "large language model" OR GPT OR ChatGPT)
STRING 2 — testing dựa spec (bổ sung baseline)
("OpenAPI specification" OR Swagger OR "API specification") AND ("test generation" OR "API testing") AND (coverage OR executable OR "fault detection" OR "bug detection")
PICOTừ khoá
P PopulationREST/Web API có OpenAPI/Swagger
I InterventionLLM (Claude Sonnet 4.6) sinh test
C Comparisonthủ công + EvoMaster (baseline)
O Outcomecoverage · executable · fault/bug detection

Vì sao 2 string: String 1 bắt các paper LLM; String 2 cố ý kéo về các baseline kinh điển (EvoMaster, Morest…) để có điểm so sánh cho GAP. Boolean: AND = bắt buộc, OR = một trong các từ, "..." = cụm chính xác.

Phần 4 · GAP-1 / 3 — COMPARISON

GAP-1 · Chưa ai so LLM ↔ thủ công ↔ EvoMaster cùng lúc

Bằng chứng:

  • LLM chỉ được so với baseline tự động: AutoRestTest & LlamaRestTest thắng EvoMaster (42 vs 20; 204 vs 130) #6,#7
  • Study #11 xếp hạng 10 tool — nhưng không tool nào kèm test người.
  • So-với-thủ-công duy nhất: EvoMaster #10 — non-LLM, và phủ thấp hơn người (41% vs 82%) #10 p.11
→ Khoảng trống
Không có đánh giá head-to-head LLM vs thiết kế test thủ công vs EvoMaster trên cùng một tập API.
?
Tại sao đây là gap thật: mỗi nhánh đã tồn tại riêng lẻ (LLM-vs-tool; tool-vs-người) nhưng chưa hợp nhất. Thiếu phép so 3 chiều → không biết LLM đã thực sự vượt người chưa, hay chỉ vượt tool.

GAP-1 = comparison. Lập luận: LLM so với tool thì có (và thắng); tool so với người thì có 1 (EvoMaster, và thua). Nhưng tam giác LLM–người–EvoMaster chưa khép.

Tại sao quan trọng: nếu LLM chỉ hơn tool mà chưa chắc hơn người, thì lời hứa "LLM thay thế test thủ công" vẫn chưa được kiểm chứng. RQ2 sẽ khép tam giác này.

Phần 4 · GAP-2 / 3 — DATASET / GROUND TRUTH

GAP-2 · Đếm lỗi mà không có sự thật nền → không tính được Recall

Bằng chứng:

  • Fault chỉ là số đếm trên hệ thống sống: 38 bug (#10), 44 bug (#12), 42 lỗi 500 (#6), 204 fault (#7), 49 crash (#8)…
  • Vì tổng số lỗi thực sự là ẩn số → không ai tính được Recall = lỗi tìm được ÷ lỗi có thật.
  • Ngoại lệ duy nhất (một phần): RestTSLLM #5 có mutation score 40.8% — nhưng trên mutant tự sinh, 6 API tự dựng, không so người/EvoMaster. #5 p.9
→ Khoảng trống
Chưa có đo lường có ground truth (trên benchmark lỗi-gieo-sẵn dùng chung) xem test LLM bắt được bao nhiêu % lỗi đã biết, so với thủ công và EvoMaster.
?
Tại sao "đếm" không đủ: "tìm 42 lỗi" vô nghĩa nếu không biết có 50 hay 500 lỗi. Chỉ khi gieo lỗi biết trước mới tính được Recall/Precision — thước đo khoa học thật sự.

GAP-2 = ground truth. Đây là gap phương pháp đo lường. Ví von: "nói tôi bắt được 42 con cá là vô nghĩa nếu không biết hồ có bao nhiêu con."

EMB có API dùng chung nhưng KHÔNG có lỗi-gieo-sẵn. Nên thí nghiệm em sẽ gieo lỗi (pre-seeded faults) để biết tổng → tính được Recall. RQ2 đóng gap này.

Phần 4 · GAP-3 / 3 — METRIC

GAP-3 · Thiếu metric edge-case & bóc tách loại endpoint bị bỏ sót

Bằng chứng:

  • Đa số chỉ đo coverage tổng + đếm 500. Chỉ #2 KAT tách 2xx/4xx #2 p.8; chỉ #3 báo 5xx.
  • Không paper nào đo số kịch bản edge-case/endpoint (4xx/5xx + boundary do cố ý vi phạm ràng buộc OpenAPI).
  • Không paper nào bóc tách coverage theo loại endpoint (CRUD / authentication / error-handling) để cho biết loại nào hay bị bỏ.
→ Khoảng trống
Không có metric chất lượng edge-case/error-code theo từng endpoint, không có hồ sơ loại endpoint bị bỏ sót, và chưa kiểm ngưỡng ≥90% endpoint trên benchmark lỗi-gieo-sẵn.
?
Tại sao cần: coverage tổng "che" việc API bỏ sót đúng các endpoint nhạy cảm (auth, error-handling). Đếm theo loại + theo kịch bản edge-case mới lộ ra điểm yếu thật. RQ1 + RQ3 đóng gap này.

GAP-3 = metric. Đây là gap về cách đo. Hai ý: (1) chưa ai đếm kịch bản edge-case theo endpoint; (2) chưa ai bóc tách theo loại endpoint.

WHY: coverage tổng có thể 90% nhưng lại bỏ đúng các endpoint auth/error — chỗ nguy hiểm nhất. RQ1 (bóc loại) + RQ3 (đếm edge-case) trả lời trực tiếp.

Phần 4 · Tuyên bố GAP hợp nhất

Phần 6 · RQ1 — Endpoint coverage

RQ1 · H0/H1 + Wilcoxon một mẫu

H0 (phủ định)
Endpoint coverage trung bình của test LLM ≤ 90% (μ ≤ 0.90)
H1 (kỳ vọng)
Endpoint coverage trung bình > 90% (μ > 0.90)
  • Test: Wilcoxon signed-rank một mẫu so coverage per-API với ngưỡng cố định 0.90 (phi tham số, mẫu nhỏ không cần giả định chuẩn).
  • Quy tắc: nếu p < 0.05 và median > 0.90 → bác H0.
  • Bổ sung: bóc tách coverage theo loại endpoint (CRUD/auth/error) → báo loại nào bị bỏ (GAP-3).
!
Ngưỡng 90% có nguồn: là target môn đặt; và đã được chứng minh khả thi (RESTifAI 95.5% operation, #9) nhưng chưa kiểm trên EMB lỗi-gieo-sẵn + chưa bóc loại.

RQ1: ngưỡng 90% endpoint. Wilcoxon một mẫu vì so với 1 ngưỡng cố định, mẫu nhỏ. Nhấn: ngưỡng có nguồn (target môn + #9 chứng minh khả thi), điểm mới = kiểm trên EMB lỗi-gieo-sẵn + bóc loại endpoint. Nguồn: hypotheses.md RQ1.

Phần 6 · RQ2 — Fault detection (gap chính)

RQ2 · LLM vs Thủ công vs EvoMaster + Friedman

H0
LLM bắt không nhiều hơn max{Thủ công, EvoMaster}
H1
LLM bắt nhiều hơn cả hai

Metric: số lỗi gieo sẵn bắt được /API (= Recall vì biết tổng).

  • Omnibus: Friedman (3 nhóm liên hệ, đo trên cùng API/lỗi).
  • Post-hoc: Wilcoxon signed-rank từng cặp (LLM-vs-tay, LLM-vs-EvoMaster) + hiệu chỉnh Holm.
  • n=3 API → bổ sung McNemar / exact binomial per-fault + effect size (vì power thấp).
!
Đây là phép so ĐẦU TIÊN đặt LLM vs thủ công vs EvoMaster trên cùng bộ lỗi-gieo-sẵn → đóng GAP-1 + GAP-2. Lý do dùng Friedman: 3 nhóm phụ thuộc, phi tham số.

RQ2 = gap chính. Friedman cho 3 nhóm phụ thuộc, rồi post-hoc Wilcoxon + Holm. Vì chỉ 3 API → thành thật về power thấp, nên thêm McNemar per-fault.

Câu chốt: "Đây là lần đầu LLM, người, EvoMaster trên cùng một bộ lỗi biết trước." Nếu hội đồng hỏi "3 API có đủ không?" → trả lời: đó là threat đã nêu, nên em bổ sung phân tích per-fault + effect size, không chỉ dựa omnibus. Nguồn: hypotheses.md RQ2.

Phần 6 · RQ3 — Edge-case scenarios

RQ3 · LLM vs Thủ công + Wilcoxon ghép cặp

H0
Số edge-case/endpoint của LLM ≤ thủ công (median)
H1
Số edge-case/endpoint của LLM > thủ công

Metric: số kịch bản 4xx/5xx + boundary riêng biệt /endpoint (invalid type, thiếu field bắt buộc, out-of-range, malformed auth…).

  • Test: Wilcoxon signed-rank ghép cặp qua các endpoint (mỗi endpoint góp 1 cặp LLM–Manual), α=0.05.
  • Quy tắc: p<0.05 và median chênh nghiêng LLM → bác H0.
  • Báo kèm bóc tách 4xx vs 5xx vs boundary.
!
Tại sao ghép cặp: mỗi endpoint là một đơn vị so sánh tự nhiên (cùng endpoint, 2 phương pháp) → kiểm soát biến nhiễu giữa các endpoint. Không paper nào đo metric này → đóng GAP-3.

RQ3: edge-case theo endpoint, ghép cặp LLM vs tay. Wilcoxon ghép cặp vì mỗi endpoint cho 2 số (LLM, tay). Đây là metric chưa ai có → đóng GAP-3. Nguồn: hypotheses.md RQ3.

Phụ lục A4

Vì sao chọn các kiểm định này + Thuật ngữ

Lý do kiểm định thống kê

  • Wilcoxon signed-rank: phi tham số, không cần giả định phân phối chuẩn — phù hợp mẫu nhỏ (RQ1 một-mẫu, RQ3 ghép-cặp).
  • Friedman (RQ2): omnibus cho 3 nhóm liên hệ (đo trên cùng API/lỗi) — không độc lập nên không dùng ANOVA/Kruskal.
  • Holm: hiệu chỉnh đa so sánh post-hoc, kiểm soát sai lầm loại I.
  • McNemar / exact binomial: per-fault nhị phân (bắt/không) — bù power thấp khi n=3 API.

Thuật ngữ

  • EMB: EvoMaster Benchmark — 20 service REST mã nguồn mở (#11).
  • Pre-seeded faults: lỗi gieo sẵn → biết tổng → tính Recall.
  • Recall: lỗi tìm được ÷ lỗi có thật.
  • Mutation score: % mutant (lỗi nhân tạo) bị test giết.
  • Endpoint vs code coverage: % endpoint gọi tới vs % dòng/nhánh code chạy.
  • Oracle: cơ chế quyết định response đúng/sai.
  • SPDG / RPG / ODG: các đồ thị mô hình hoá phụ thuộc operation.

Slide tra cứu khi bị hỏi sâu về thống kê hoặc thuật ngữ. Điểm mấu chốt: chọn test phi tham số vì mẫu nhỏ + nhóm phụ thuộc; thành thật bù power bằng per-fault analysis.
