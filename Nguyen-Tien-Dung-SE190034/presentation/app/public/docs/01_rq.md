## 1. Initial RQ (Stage 0 — before PRISMA)

> *"For a REST API with an OpenAPI specification (P), can an LLM automatically generate API test cases that cover edge cases and error codes (I), compared to manual test design (C), and reach endpoint coverage ≥ 90% and detect at least 2 additional real bugs (O)?"*

**2 ambiguous points → why PRISMA is needed:**
1. **[Which specific LLM?]** — the initial RQ only says "LLM" in general; it is not yet known which model has / has not been evaluated for REST API test generation.
2. **[Which baseline & where does the threshold come from?]** — the initial RQ only compares against "manual test design"; it is unclear whether an automated baseline tool should be added, and whether the coverage threshold ≥ 90% / "≥ 2 real bugs" has a source or is self-set.

## 3. Official RQ (after PRISMA — refined)

### Main RQ
> **"How effectively can LLMs automatically generate API test cases from OpenAPI specifications, compared to manually designed tests and EvoMaster-generated tests on pre-seeded faulty API versions?"**

### RQ1 (Coverage — targets GAP-M)
> **"Do LLM-generated API tests achieve endpoint coverage ≥ 90% on locally deployed EvoMaster benchmark APIs, and which endpoint types (CRUD, authentication, error handling) are most frequently missed?"**

### RQ2 (Fault detection — targets GAP-D)
> **"How many pre-seeded faults do LLM-generated tests detect compared to manually designed tests and EvoMaster-generated tests across 3 benchmark APIs?"**

### RQ3 (Error codes & boundary — targets GAP-M)
> **"Are LLM-generated tests more effective than manual test design at producing test cases for error codes (4xx, 5xx) and boundary conditions, measured by edge case scenario count per endpoint?"**

---

## 4. PICO for the experiment

- **P (Population):** 3 REST API services taken from the **EvoMaster Benchmark (EMB)** that ship with an OpenAPI/Swagger spec, deployed locally, and instrumented with **pre-seeded faulty versions** so that the total number of faults (ground truth) is known.
- **I (Intervention):** An automated API-test-case generator built on an **LLM** (reads the OpenAPI spec → generates positive / negative / error-code / boundary test cases). Output is a tool + GitHub repo.
- **C (Comparison):** (1) **Manual test design** (tests written by hand); (2) **EvoMaster** (search-based automated test generation, the most common baseline tool in the evidence table).
- **O (Outcome):**
  - **O1 — Endpoint coverage (%):** the proportion of endpoints exercised, target threshold **≥ 90%**; plus a breakdown of missed endpoint types (CRUD / authentication / error-handling).
  - **O2 — Fault detection:** the number of pre-seeded faults detected, comparing LLM vs manual vs EvoMaster across the 3 APIs.
  - **O3 — Edge-case scenario count:** the number of error-code (4xx/5xx) + boundary-condition scenarios per endpoint (LLM vs manual).
